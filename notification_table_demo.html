🗄️ Database Schema

The notification system uses a comprehensive SQLite database schema with the following structure:

Column Type Description
id INTEGER PRIMARY KEY Auto-incrementing unique identifier
user_id TEXT NOT NULL Firebase UID or user identifier
title TEXT NOT NULL Notification title/subject
message TEXT NOT NULL Notification content/body
type TEXT Visual type: info, success, warning, error, alert
category TEXT Functional category: medical, appointment, system, etc.
priority TEXT Priority level: low, normal, high, urgent
is_read INTEGER Read status (0=unread, 1=read)
metadata TEXT JSON field for additional context data
created_at TEXT Timestamp when notification was created

🔌 API Endpoints

GET /api/notifications
Retrieve user notifications with pagination and filtering
POST /api/notifications
Create a new notification
PUT /api/notifications/{id}
Update notification (mark as read, archive, etc.)
DELETE /api/notifications/{id}
Delete a notification
GET /api/notifications/stats
Get notification statistics for a user
POST /api/notifications/bulk
Perform bulk operations (mark all as read, etc.)

✨ Key Features

🔄 Real-time Updates

Notifications are updated in real-time with automatic refresh and push capabilities.

🎯 Priority System

Four priority levels (low, normal, high, urgent) with visual indicators and sorting.

🏷️ Smart Categorization

Organized by category (medical, appointment, system, medication) for easy filtering.

📊 Analytics & Stats

Comprehensive statistics tracking for notification engagement and patterns.

🔍 Advanced Filtering

Filter by read/unread status, category, priority, and date ranges.

⚡ Bulk Operations

Mark all as read, delete multiple, and other batch operations for efficiency.
